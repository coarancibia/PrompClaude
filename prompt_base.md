# Prompt base — Proyecto PrompClaude

Estás trabajando sobre el proyecto `PrompClaude`, un portal interno de prompts, skills e instrucciones para Product Designers.

## Objetivo del proyecto

El objetivo del sitio es ayudar al equipo de Product Design a trabajar de forma más ordenada, rápida y consistente usando Claude Code como apoyo.

Este proyecto no busca reemplazar el criterio del Product Designer ni del UX/UI Designer. Su función es apoyar tareas repetitivas, operativas, de documentación, revisión y automatización del flujo diario de diseño.

## Reglas generales

Antes de modificar cualquier archivo:

1. Revisa la estructura actual del proyecto.
2. Identifica qué archivos necesitas tocar.
3. Mantén el estilo visual actual del sitio.
4. No elimines contenido existente sin avisarme.
5. No cambies el nombre de los archivos.
6. No crees archivos nuevos si no es necesario.
7. No rompas la navegación interna.
8. Mantén consistencia visual entre todas las páginas.
9. Mantén el responsive funcionando correctamente.
10. Aplica los cambios solo donde corresponda.

## Estilo visual del sitio

Mantén el look & feel actual:

- Fondo grafito / dark.
- Cards con efecto premium.
- Pills o badges de estado.
- Navegación clara.
- Jerarquía visual escaneable.
- Layout ordenado.
- Estilo moderno, editorial y profesional.
- Experiencia tipo portal interno para el equipo.

## Archivos principales

Trabaja principalmente sobre estos archivos:

- `index.html`
- `agente.html`
- `componentes.html`
- `naming.html`
- `auditoria.html`

No cambies estos nombres.

## Criterios de contenido

El contenido debe ser claro, accionable y fácil de copiar por el equipo.

Evita textos demasiado genéricos. Cada sección debe explicar:

- Para qué sirve.
- Cuándo usarlo.
- Qué input necesita.
- Qué output entrega.
- Cómo se usa en el flujo real de trabajo.
- Qué cuidado debe tener el Product Designer al usarlo.

## Enfoque de agente y skills

Cuando trabajes en `agente.html`, recuerda que el enfoque correcto es:

Un agente principal de Product Design que puede activar distintos skills según la tarea.

No presentar como varios agentes independientes.

Los skills deben ayudar a automatizar o acelerar tareas como:

- Síntesis de research.
- Síntesis de hallazgos post-test.
- Mapeo de flujos, estados y casuísticas.
- Handoff a desarrollo.
- Criterios de aceptación para Jira / QA.
- Framing del problema.
- Revisión UX/UI.
- Redacción UX.
- Benchmark / exploración de conceptos.
- Síntesis de reunión.
- Documentación de decisiones.
- Preparación para critique o revisión con stakeholders.

## Relación con recursos existentes

Cuando trabajes la parte de UX/UI, conecta con los recursos ya existentes:

- `componentes.html` para crear o revisar componentes.
- `naming.html` para nombrar pantallas.
- `auditoria.html` para auditoría visual o auditoría DS.

Esta parte debe quedar como apoyo para revisar, ordenar y estandarizar la capa UI antes de critique, handoff o QA.

## Antes de aplicar cambios

Antes de modificar, dime brevemente:

- Qué entendiste.
- Qué archivos vas a tocar.
- Qué cambio vas a aplicar.

Luego aplica el cambio.

## Después de aplicar cambios

Al terminar, revisa:

- Que los links internos funcionen.
- Que el diseño siga consistente.
- Que no se haya roto el responsive.
- Que no se haya eliminado contenido importante.
- Que el cambio se entienda como apoyo al Product Designer, no como reemplazo.
