01

Contexto general del proyecto

El naming usa el mismo proyecto de Claude que ya tiene cargado el contexto de flujos, marca y forma de trabajo. No se crea un proyecto nuevo: se agrega una regla adicional sobre el que ya existe.

Qué debe tener el proyecto
Instrucciones del proyecto
El proyecto debe contener: contexto de estilos y componentes existentes, flujos del producto, lineamientos de marca y la forma de trabajo del equipo. El modo de naming (bloque 02) se agrega como una regla adicional, no como un proyecto nuevo.
02

Modo naming de pantallas

Este bloque se agrega al final de las instrucciones del proyecto. Define la nomenclatura y le dice a Claude que trabaje solo sobre el link indicado, sin tocar diseño ni contenido visual.

Bloque para instrucciones del proyecto
Instrucciones del proyecto · pegar al final
Modo naming de pantallas en Figma:

Cuando pida nombrar pantallas y entregue un link de Figma, renombrar directamente los frames/pantallas del link indicado.

Aplicar esta nomenclatura:

proyecto/dispositivo/flujo/pantalla_o_estado/foco_ajuste

Reglas:
- Trabajar solo sobre el link, frame o sección indicada.
- No cambiar diseño, componentes ni contenido visual.
- No leer todo el archivo si no es necesario.
- Usar nombres claros, entendibles y orientados al flujo.
- No usar nombres técnicos tipo node_123.
- Usar minúsculas, sin tildes y con snake_case.
- Mantener consistencia entre pantallas del mismo flujo.
- Si falta contexto, inferir el nombre más claro según la pantalla.

Ejemplo:
cencoprimeblack/desktop/gestion_cuenta/skeleton/carga_inicial

No hacer explicación larga. Priorizar renombrar en Figma.
Se pega una sola vez, al final de las instrucciones del proyecto. No se repite en los chats.

Mantén la nomenclatura:
proyecto/dispositivo/flujo/pantalla_o_estado/foco_ajuste

08

Checklist de revisión

Después de que Claude nombre las pantallas, esto se revisa manualmente antes de dar por terminada la tarea.

Estructura

  • El nombre sigue la estructura definida.
  • Tiene proyecto, dispositivo, flujo, pantalla/estado y foco.
  • Usa / para separar niveles.
  • Usa snake_case.

Claridad

  • El nombre se entiende sin abrir el frame.
  • No usa nombres genéricos como "pantalla_1" o "final".
  • No usa nombres técnicos tipo node_123.
  • Se entiende el flujo al que pertenece.

Consistencia

  • Todas las pantallas del flujo tienen la misma lógica.
  • El dispositivo está bien indicado.
  • El proyecto está escrito igual en todos los frames.
  • Los estados se nombran de forma consistente.
09

Buenas prácticas para ahorrar tokens

Reglas simples para que el equipo trabaje rápido sin gastar contexto repitiendo lo que el proyecto ya tiene configurado.

Hacer

  • Usar un chat exclusivo para naming.
  • No repetir todo el contexto en cada mensaje.
  • Entregar solo link, proyecto, dispositivo y flujo.
  • Usar nombres de flujo claros.
  • Pedir ajustes sobre el mismo link si algo queda mal.

Evitar

  • Pedir "antes de comenzar dime lo que harás".
  • Pedir explicación larga.
  • Mandar links de todo el archivo si solo se debe trabajar una sección.
  • Mezclar naming, componentes, documentación y revisión UI en el mismo chat.
  • Usar nombres con tildes, mayúsculas o espacios.
