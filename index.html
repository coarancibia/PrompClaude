Instructivo de prompts para el equipo

Guía base para trabajar con Claude y mantener criterios consistentes en diseño, documentación y revisión de flujos mientras avanzamos hacia el uso de agentes.

Esta guía no reemplaza el criterio de diseño, pero ayuda a ordenar la forma en que pedimos, revisamos y documentamos.

Cómo usar un prompt

Los prompts base de esta guía están pensados para adaptarse, no para copiarse tal cual.

1
Copia el prompt base que más se acerque a lo que necesitas.
2
Reemplaza los campos entre corchetes por el nombre real de la pantalla, archivo o flujo.
3
Agrega contexto del flujo o pantalla que Claude no pueda deducir por sí solo.
4
Indica restricciones visuales, funcionales o de contenido si aplican.
5
Pide primero análisis, antes de pedir ejecución. Separa siempre estos dos pasos.
6
Revisa el resultado antes de aplicarlo o de darlo por terminado.
Estructura recomendada de un prompt

Todo prompt sólido puede armarse con estos ocho bloques. No todos son obligatorios, pero conviene tenerlos presentes.

Contexto
Qué es la pantalla o flujo y en qué estado está hoy.
Objetivo
Qué se quiere lograr con esta tarea puntual.
Archivo o pantalla a trabajar
Nombre o ruta exacta de lo que Claude debe abrir.
Restricciones
Qué no se puede hacer: no rediseñar, no inventar componentes, etc.
Criterios de diseño
Qué componentes, estilos o patrones ya existentes debe respetar.
Criterios de accesibilidad
Nivel esperado, por ejemplo AA, y qué validar puntualmente.
Entregable esperado
Qué formato de respuesta se espera: lista, checklist, comparación.
Qué no debe modificar
Qué queda explícitamente fuera del alcance de esta tarea.
Prompts base para copiar

Copia el bloque, reemplaza lo que está entre corchetes y agrega el contexto que falte.

Revisar una pantalla a nivel UI/UX

Para pedir un diagnóstico priorizado antes de tocar nada.

Contexto: estoy trabajando en [nombre de la pantalla o flujo].
Objetivo: revisar la pantalla a nivel UI/UX sin modificar nada todavía.
Archivo a analizar: [ruta o nombre del archivo].
Restricciones: no ejecutes cambios, solo entrega un análisis.
Evalúa: jerarquía visual, consistencia con el resto del flujo, claridad de la información y usabilidad general.
Entregable esperado: hallazgos priorizados (crítico / alto / medio / bajo) con una breve explicación de cada uno.
Prompt

Mejorar una pantalla sin cambiar estructura

Para pulir detalle visual sin tocar el layout ni el orden de bloques.

Contexto: [nombre de la pantalla] ya tiene una estructura definida y funciona bien a nivel de layout.
Objetivo: mejorar el detalle visual (espaciado, jerarquía, contraste, textos) sin modificar la estructura ni el orden de los bloques.
Archivo a trabajar: [archivo].
Restricciones: no cambies la estructura, no agregues ni elimines secciones, no inventes componentes nuevos.
Criterios de diseño: usa los componentes y estilos ya definidos en el proyecto.
Entregable esperado: los mismos bloques, con mejoras puntuales y una lista de qué cambiaste y por qué.

Documentar un flujo

Para dejar un flujo entendible por cualquier persona del equipo.

Contexto: necesito documentar el flujo de [nombre del flujo] para que cualquier persona del equipo lo entienda.
Objetivo: generar documentación clara del flujo, paso a paso.
Archivo o pantallas a revisar: [archivo(s) o pantallas del flujo].
Entregable esperado: descripción general, pasos del flujo, estados posibles (vacío, error, carga, éxito) y casos borde a considerar.
Qué no debe modificar: no cambies el código ni el diseño, solo documenta lo que existe.
Revisión

Auditar consistencia visual

Para comparar una pantalla contra el resto del proyecto.

Contexto: quiero validar que [pantalla o flujo] sea consistente con el resto del proyecto.
Objetivo: auditar el uso de estilos, espaciados, tipografía y componentes.
Archivo a analizar: [archivo].
Restricciones: no ejecutes cambios, solo reporta.
Compara contra: [archivo o pantalla de referencia].
Entregable esperado: listado de inconsistencias priorizadas (crítico / alto / medio / bajo).
Checklist

Revisar accesibilidad básica

Para validar contraste, foco, orden de lectura y etiquetas.

Contexto: quiero validar accesibilidad básica en [pantalla o flujo].
Objetivo: revisar contraste de color, tamaños de texto, foco visible, orden de lectura y etiquetas.
Archivo a analizar: [archivo].
Criterios de accesibilidad: nivel AA como mínimo.
Restricciones: no ejecutes cambios todavía, solo entrega el diagnóstico.
Entregable esperado: checklist de accesibilidad con hallazgos y nivel de severidad.
Checklist

Preparar handoff para desarrollo

Para verificar que una pantalla está lista antes de pasarla a dev.

Contexto: [pantalla o flujo] está por pasar a desarrollo.
Objetivo: preparar el handoff antes de entregar al equipo de desarrollo.
Archivo a revisar: [archivo].
Verifica: naming consistente, estados definidos, componentes usados correctamente y accesibilidad básica.
Entregable esperado: checklist de handoff con lo que está listo y lo que falta antes de entregar.
Qué no debe modificar: no cambies el diseño, solo evalúa si está listo para entregar.
Revisión

Revisar uso de componentes existentes

Para detectar componentes duplicados o mal usados.

Contexto: quiero asegurarme de que [pantalla o flujo] use los componentes ya definidos en el proyecto.
Objetivo: identificar dónde se usan componentes fuera del set definido o se duplica algo que ya existe.
Archivo a analizar: [archivo].
Restricciones: no inventes componentes nuevos si ya existe uno que cumple la función.
Entregable esperado: componentes usados correctamente, mal usados o duplicados, con sugerencia de reemplazo.
Checklist

Generar checklist de revisión

Para dejar una lista accionable antes de dar algo por terminado.

Contexto: necesito un checklist de revisión para [pantalla o flujo] antes de darlo por terminado.
Objetivo: generar un checklist claro y accionable.
Archivo a revisar: [archivo].
Entregable esperado: checklist con ítems verificables (sí/no), agrupados por diseño, contenido, accesibilidad y consistencia.
Prompt

Mejorar textos de una pantalla

Para pulir redacción sin tocar estructura ni diseño.

Contexto: los textos de [pantalla] necesitan revisión.
Objetivo: mejorar claridad, tono y consistencia de los textos sin cambiar la estructura ni el diseño.
Archivo a trabajar: [archivo].
Restricciones: no cambies la función de cada texto (título, ayuda, error, etc.), solo mejora la redacción.
Entregable esperado: lista de textos actuales versus propuesta, con una breve razón del cambio.
Revisión

Comparar versión actual versus nueva propuesta

Para tomar decisiones con un análisis objetivo, no solo visual.

Contexto: tengo dos versiones de [pantalla o flujo]: la actual y una nueva propuesta.
Objetivo: comparar ambas versiones de forma objetiva.
Archivos a comparar: [archivo actual] y [archivo nuevo].
Entregable esperado: comparación con diferencias, qué mejora, qué empeora y una recomendación final.
Qué no debe modificar: no elijas por mí, solo entrega el análisis comparativo.
Reglas para obtener resultados consistentes
→ Siempre indica el objetivo del flujo, no solo la tarea puntual.
→ No pidas cambios visuales sin explicar cuál es el problema que resuelven.
→ Separa siempre análisis de ejecución: primero que revise, después que ejecute.
→ Pide que no invente componentes si ya existen componentes definidos.
→ Pide que mantenga la estructura cuando el objetivo es solo una mejora visual.
→ Indica siempre si el resultado es para desktop, mobile o responsive.
→ Pide que explique sus decisiones cada vez que haga un cambio.
Mientras esperamos los agentes

Una guía temporal, no una versión final.

Hoy esta página funciona como una guía editable y centralizada para ordenar cómo el equipo le pide tareas a Claude: qué contexto dar, qué restricciones poner y qué entregable esperar.

Cuando existan agentes especializados por tipo de tarea (revisión, documentación, naming, accesibilidad), buena parte de estos prompts se van a mover ahí. Hasta entonces, esta guía es la fuente de verdad compartida del equipo.
