01

Modo auditoría del DS

Este bloque se agrega al final de las instrucciones del proyecto. Define qué debe revisar Claude, cómo debe entregar los hallazgos y qué no debe tocar.

Bloque para instrucciones del proyecto
Instrucciones del proyecto · pegar al final
Modo auditoría de uso del DS en Figma:

Cuando pida auditar un Figma y entregue un link, revisar si las pantallas están usando correctamente el DS Tokens y DS Components existentes.

Objetivo:
Detectar si el diseño está bien construido según el Design System o si existen inconsistencias, componentes duplicados, estilos manuales, tokens mal aplicados, problemas de accesibilidad, responsive o layout.

Revisar principalmente:
- Uso correcto de componentes existentes del DS.
- Componentes creados manualmente cuando ya existe uno en la librería.
- Instancias detachadas o modificadas fuera de la lógica del componente.
- Variantes/properties mal usadas.
- Colores hardcodeados en vez de tokens.
- Tipografías fuera del DS.
- Spacing, padding, gap, radius y tamaños fuera de tokens.
- Estados faltantes o mal aplicados: default, hover, pressed, focus, disabled, error, selected, loading, empty.
- Auto layout, constraints, Hug contents y Fill container.
- Responsive y adaptación al contenedor.
- Accesibilidad AA: contraste, legibilidad, tamaño táctil, jerarquía visual y foco visible.
- Naming de frames, componentes o capas si afecta la claridad del handoff.

Reglas de trabajo:
- Trabajar solo sobre el link, frame o sección indicada.
- No modificar diseño, componentes ni contenido visual, salvo que se pida explícitamente.
- No leer todo el archivo si no es necesario.
- Comparar contra DS Tokens y DS Components como fuente de verdad.
- Si existe un componente del DS que debería usarse, indicarlo.
- Si falta una regla, token o componente en el DS, marcarlo como "pendiente de definición".
- No hacer una explicación larga. Entregar hallazgos claros, accionables y priorizados.

Formato de entrega:
- Resumen general del estado del uso del DS.
- Hallazgos por prioridad: crítico, alto, medio, bajo.
- Para cada hallazgo indicar:
  - Pantalla o sección.
  - Problema detectado.
  - Impacto en experiencia, consistencia, accesibilidad o desarrollo.
  - Recomendación concreta.
  - Componente/token del DS que debería usarse, si aplica.
Se pega una sola vez al final de las instrucciones del proyecto.

Corregir hallazgos después de auditar

Primero se audita, el equipo decide qué corregir, y después se pide la corrección. No mezclar ambas tareas en el mismo mensaje.

Prompt de corrección
Chat "Auditoría DS Figma" · después de revisar hallazgos
Corregir hallazgos de la auditoría DS:

Link:
[LINK FIGMA]

Corregir solo los hallazgos de prioridad:
[crítico/alto/medio/bajo]

Usar DS Tokens y DS Components existentes.

No rediseñar el flujo. Solo ajustar inconsistencias detectadas en la auditoría.
La segunda línea es la única que cambia. Se puede pedir corrección de una o varias prioridades a la vez.
05

Qué revisa Claude en una auditoría

Esta es la lista completa de lo que Claude chequea, organizada por área. Sirve como referencia para saber qué esperar de la auditoría.

Componentes

  • Componentes existentes del DS bien usados.
  • Elementos manuales que deberían ser componentes.
  • Componentes duplicados.
  • Instancias detachadas sin justificación.
  • Variantes/properties bien utilizadas.
  • Estados faltantes del componente.

Tokens

  • Colores aplicados desde tokens (no hardcodeados).
  • Tipografía del DS.
  • Spacing, padding y gap con tokens.
  • Radius correcto.
  • Overlays según el DS.
  • Tamaños de íconos y jerarquía visual.

Layout y responsive

  • Auto Layout correcto.
  • Hug contents y Fill container bien usados.
  • Constraints configurados.
  • Elementos dentro del frame.
  • Comportamiento responsive correcto.
  • Sin anchos fijos innecesarios.

Accesibilidad AA

  • Contraste mínimo AA.
  • Legibilidad del texto.
  • Tamaños táctiles adecuados.
  • Foco visible.
  • Jerarquía de información clara.
  • Estados de error no dependen solo del color.

Handoff

  • Nombres claros de frames y capas.
  • Capas entendibles para desarrollo.
  • Frames ordenados.
  • Componentes fáciles de identificar.
  • Nomenclatura consistente si aplica.

Estados visuales

  • Default, hover, pressed, focus.
  • Disabled, error, selected.
  • Loading y empty states.
  • Todos los estados necesarios presentes.
  • Estados aplicados de forma consistente.

07

Criterios de prioridad

Lógica que usa Claude para ordenar los hallazgos. Sirve también para que el equipo decida qué corregir primero.

PrioridadCuándo usarla
Crítico Rompe el uso del DS, afecta accesibilidad, bloquea desarrollo o daña la experiencia principal.
Alto Genera inconsistencia importante o puede afectar responsive y handoff.
Medio Es una mejora relevante, pero no bloquea el flujo ni la experiencia principal.
Bajo Ajuste menor de orden, naming, limpieza o consistencia visual secundaria.
08

Buenas prácticas para ahorrar tokens

Reglas para que el equipo trabaje rápido sin gastar contexto repitiendo lo que el proyecto ya tiene configurado.

Hacer

  • Usar un chat exclusivo para auditoría DS.
  • Entregar solo link, proyecto, dispositivo y flujo.
  • Pedir auditoría sobre un frame o sección específica.
  • Separar auditoría de corrección en mensajes distintos.
  • Pedir correcciones solo después de revisar los hallazgos.
  • Priorizar hallazgos accionables antes de pedir que corrija.

Evitar

  • Mandar el archivo completo si solo se revisa una sección.
  • Pedir análisis muy largo o narrativo.
  • Mezclar auditoría con creación de componentes.
  • Mezclar auditoría con naming de pantallas.
  • Pedir que corrija mientras audita, salvo que sea necesario.
  • Repetir todo el contexto del DS si ya está en el proyecto.
